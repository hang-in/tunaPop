# Phase 12 + 14 + 15 — Release readiness bundle

## Phase

Three v1 release-gate phases bundled into one PR:

- **Phase 12**: `PrivacyInfo.xcprivacy` manifest (required for
  notarization and macOS 15+ submission readiness).

## Focus

Make tunaPop a signed, notarized, auto-updatable macOS app you can
ship from GitHub Releases. Add the Privacy manifest required by
Apple. Wire Sparkle so users get over-the-air updates. Stand up
GitHub Actions so the release process is reproducible.

The Swift Package is the source-of-truth target. The `.app` bundle
is built via a release script that wraps `swift build -c release`,
copies the executable, embeds Info.plist + PrivacyInfo +
Resources, code-signs, and notarizes.

Files to add:
- `Sources/TunaPop/Resources/PrivacyInfo.xcprivacy`
- `Sources/TunaPop/Resources/Info.plist`
- `scripts/package.sh` — builds the `.app`, signs, notarizes,
  staples, produces a DMG
- `scripts/sparkle-sign.sh` — signs the DMG for Sparkle EdDSA
- `.github/workflows/build.yml` — PR / push CI
- `.github/workflows/lint.yml` — SwiftLint / SwiftFormat dry-run
- `.github/workflows/release.yml` — tag-triggered release pipeline
- `.swiftlint.yml` — SwiftLint configuration
- `.swiftformat` — SwiftFormat configuration
- `appcast.xml` — Sparkle appcast template (committed to repo;
  release.yml updates it on each release)
- `homebrew/Casks/tunapop.rb` — Homebrew Cask formula template

Files to modify:
- `Package.swift` — add Sparkle dependency, declare Resources path
- `Sources/TunaPop/TunaPopApp.swift` — wire `SPUStandardUpdater`
- `Sources/TunaPop/SettingsView.swift` — "업데이트 확인" button in
  "고급" section
- `docs/MASTER_SPEC.md` §15.1 — document new UserDefaults keys
  added by Sparkle (e.g. `SUEnableAutomaticChecks`)

Files NOT to modify:
- Anything in `Sources/TunaPop/` other than `TunaPopApp.swift` and
  `SettingsView.swift`.

## Constraints

- macOS 14+, Swift 6.0+ (per `Package.swift`).
- Sparkle 2.x is the ONE allowed new dependency. Everything else
  stays native macOS.
- `swift build` MUST succeed with zero new warnings.
- The release pipeline MUST gracefully handle missing GitHub
  Secrets (skip signing / notarization steps when secrets are
  absent — surface a clear warning rather than failing the job).
- Bundle identifier: `app.tunapop` (per master spec §21 decision).
- Marketing version source: `Bundle.appVersionDisplay` continues to
  read `CFBundleShortVersionString` from `Info.plist`.
- Notarization requires a Developer ID Application certificate
  available to the runner. CI uses GitHub Actions secrets;
  manual `scripts/package.sh` reads from `~/.tunapop-release-env`
  (gitignored).

## Part 12 — `PrivacyInfo.xcprivacy`

Apple requires apps to declare third-party SDK usage and "required
reason" API usage for notarized binaries (and especially App Store
submissions). v1 ships a minimal manifest.

`Sources/TunaPop/Resources/PrivacyInfo.xcprivacy`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>NSPrivacyTracking</key>
    <false/>
    <key>NSPrivacyTrackingDomains</key>
    <array/>
    <key>NSPrivacyCollectedDataTypes</key>
    <array/>
    <key>NSPrivacyAccessedAPITypes</key>
    <array>
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>CA92.1</string>
            </array>
        </dict>
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategorySystemBootTime</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>35F9.1</string>
            </array>
        </dict>
    </array>
</dict>
</plist>
```

- `CA92.1`: "Declare this reason to access user defaults to read
  and write information that is only accessible to the app itself."
- `35F9.1`: "Declare this reason to access the system boot time in
  order to measure the amount of time that has elapsed between
  events that occurred within the app." Used implicitly by macOS
  unified logging.

If the binary turns out not to access SystemBootTime in v1, the
entry is harmless — but keep the audit honest: confirm with
`xcrun otool -L .build/release/TunaPop` whether `libsystem_kernel`
or `Foundation`'s boot-time helpers are linked. If not, drop the
35F9.1 entry. (Default: keep both.)

### `.github/workflows/release.yml`

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  release:
    runs-on: macos-latest
    steps:
      - uses: actions/checkout@v4
      - name: Select Xcode 16
        run: sudo xcode-select -s /Applications/Xcode_16.app
      - name: Import signing certificate
        if: ${{ secrets.MAC_CERTIFICATE_BASE64 != '' }}
        env:
          MAC_CERTIFICATE_BASE64: ${{ secrets.MAC_CERTIFICATE_BASE64 }}
          MAC_CERTIFICATE_PASSWORD: ${{ secrets.MAC_CERTIFICATE_PASSWORD }}
        run: scripts/import-cert.sh
      - name: Package and notarize
        env:
          DEVELOPER_ID: ${{ secrets.DEVELOPER_ID_APPLICATION }}
          APPLE_ID: ${{ secrets.APPLE_ID }}
          APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
          APP_PASSWORD: ${{ secrets.APP_SPECIFIC_PASSWORD }}
        run: scripts/package.sh
      - name: Sign for Sparkle
        env:
          SPARKLE_PRIVATE_KEY: ${{ secrets.SPARKLE_PRIVATE_KEY }}
        if: ${{ env.SPARKLE_PRIVATE_KEY != '' }}
        run: scripts/sparkle-sign.sh
      - name: Update appcast
        env:
          VERSION: ${{ github.ref_name }}
        run: scripts/update-appcast.sh
      - name: Upload to release
        uses: softprops/action-gh-release@v2
        with:
          files: |
            dist/tunaPop-*.dmg
            appcast.xml
```

### Required GitHub Secrets

(Document in `docs/RELEASE.md` — separate user-facing doc.)

- `MAC_CERTIFICATE_BASE64`: base64-encoded `.p12` of Developer ID
  Application certificate
- `MAC_CERTIFICATE_PASSWORD`: password for the `.p12`
- `DEVELOPER_ID_APPLICATION`: certificate common name, e.g.
  `"Developer ID Application: Your Name (TEAMID)"`
- `APPLE_ID`: Apple ID email
- `APPLE_TEAM_ID`: 10-character team ID
- `APP_SPECIFIC_PASSWORD`: app-specific password for `notarytool`
- `SPARKLE_PRIVATE_KEY`: EdDSA private key generated by Sparkle's
  `generate_keys` tool. Public key goes into `Info.plist`.

### `scripts/package.sh`

```bash
#!/bin/bash
set -euo pipefail

VERSION=${GITHUB_REF_NAME#v}
VERSION=${VERSION:-0.1.0-dev}

# Build release binary
swift build -c release --arch arm64 --arch x86_64

# Bundle skeleton
APP="dist/tunaPop.app"
rm -rf "$APP"
mkdir -p "$APP/Contents/MacOS" "$APP/Contents/Resources"

cp .build/apple/Products/Release/TunaPop "$APP/Contents/MacOS/tunaPop"
cp Sources/TunaPop/Resources/Info.plist "$APP/Contents/Info.plist"
cp Sources/TunaPop/Resources/PrivacyInfo.xcprivacy "$APP/Contents/Resources/PrivacyInfo.xcprivacy"

# Update versions in Info.plist
/usr/libexec/PlistBuddy -c "Set :CFBundleShortVersionString $VERSION" "$APP/Contents/Info.plist"
/usr/libexec/PlistBuddy -c "Set :CFBundleVersion $(date +%s)" "$APP/Contents/Info.plist"

# Code sign
if [ -n "${DEVELOPER_ID:-}" ]; then
  codesign --deep --force --options runtime \
    --sign "$DEVELOPER_ID" \
    --entitlements scripts/entitlements.plist \
    "$APP"
fi

# Notarize
if [ -n "${APPLE_ID:-}" ] && [ -n "${APP_PASSWORD:-}" ]; then
  DMG="dist/tunaPop-$VERSION.dmg"
  hdiutil create -volname "tunaPop" -srcfolder "$APP" -ov -format UDZO "$DMG"
  xcrun notarytool submit "$DMG" \
    --apple-id "$APPLE_ID" \
    --team-id "$APPLE_TEAM_ID" \
    --password "$APP_PASSWORD" \
    --wait
  xcrun stapler staple "$DMG"
else
  # Fallback: just DMG without notarization
  DMG="dist/tunaPop-$VERSION.dmg"
  hdiutil create -volname "tunaPop" -srcfolder "$APP" -ov -format UDZO "$DMG"
fi
```

### `scripts/import-cert.sh`

```bash
#!/bin/bash
set -euo pipefail
KEYCHAIN="build.keychain"
echo "$MAC_CERTIFICATE_BASE64" | base64 --decode > /tmp/cert.p12
security create-keychain -p actions "$KEYCHAIN"
security default-keychain -s "$KEYCHAIN"
security unlock-keychain -p actions "$KEYCHAIN"
security import /tmp/cert.p12 -k "$KEYCHAIN" -P "$MAC_CERTIFICATE_PASSWORD" -T /usr/bin/codesign
security set-key-partition-list -S apple-tool:,apple:,codesign: -s -k actions "$KEYCHAIN"
rm /tmp/cert.p12
```

## Acceptance Criteria

1. `swift build` succeeds with zero new warnings (Sparkle pulled in
   as a dependency).
2. New files exist per the list above:
   - `Sources/TunaPop/Resources/PrivacyInfo.xcprivacy`
   - `Sources/TunaPop/Resources/Info.plist`
   - `scripts/package.sh`, `scripts/entitlements.plist`,
     `scripts/import-cert.sh`
   - `.github/workflows/build.yml`, `lint.yml`, `release.yml`
   - `.swiftlint.yml`, `.swiftformat`
   - `appcast.xml`
   - `homebrew/Casks/tunapop.rb`
3. `Package.swift` declares the Sparkle dependency and exposes
   `Resources/Info.plist`, `Resources/PrivacyInfo.xcprivacy` to the
   bundle.
4. `TunaPopApp.AppDelegate` owns `SPUStandardUpdaterController` with
   `startingUpdater: true`. The `updater` property is accessible
   from `SettingsView`.
5. Settings "고급" section shows the "업데이트 확인..." button above
   the verbose toggle. Clicking it triggers
   `updater.checkForUpdates()` (results visible via Sparkle's
   built-in modal — no app code needed).
6. Running `bash scripts/package.sh` locally with empty env vars:
   produces `dist/tunaPop.app` and `dist/tunaPop-0.1.0-dev.dmg`
   WITHOUT codesigning or notarization (gracefully skips).
7. With real Developer ID env vars: the produced DMG is signed,
   notarized, stapled. Verified with
   `spctl -a -t open --context context:primary-signature dist/tunaPop-*.dmg`.
8. GitHub Actions workflows are committed and pass on a stub PR
   (build.yml runs `swift build` clean; lint.yml runs SwiftLint
   strict with no errors after Phase 10's logging cleanup).
9. `release.yml` is gated on `v*` tags. Pushing `v0.1.0` triggers
   the pipeline. With all secrets set, a notarized DMG appears as a
   GitHub Release asset.
10. The Privacy manifest validates against Apple's schema (XML
    parse + Apple's privacy-report tooling in Xcode shows the
    correct categories).
11. Bundle identifier in the produced `.app` is `app.tunapop`.
    `LSUIElement = true` means no Dock icon.
12. Sparkle's public key in `Info.plist` is a placeholder until
    the maintainer generates the EdDSA pair. The signing step in
    `release.yml` is conditional on the secret being set.
13. The Homebrew Cask template references the canonical release
    URL pattern but does not need to be valid Brew syntax until
    the first publish.

## macOS edge-case checklist (Appendix C)

- [ ] Permissions: Info.plist `NSAccessibilityUsageDescription` +
      `NSAppleEventsUsageDescription` strings are present. Users see
      these strings in the macOS permission prompt instead of the
      default macOS-supplied generic text.
- [ ] Permission revoked at runtime: no change.
- [ ] Key window: no change.
- [ ] Z-order: no change.
- [ ] Animation anchor: no change.
- [ ] Mouse / key event routing: no change.
- [ ] Cancellation: Sparkle's update dialogs run in a separate
      modal. Our cancellation paths are unaffected.
- [ ] Resource cleanup: `SPUStandardUpdaterController` lifetime
      tied to AppDelegate.
- [ ] UserDefaults schema: ADDS Sparkle keys
      (`SUEnableAutomaticChecks`, `SULastCheckTime`,
      `SUHasLaunchedBefore`, etc.) automatically. Document in
      master spec §15.1.

## Out of Scope

- Hosting the appcast feed at a real domain — that's an
  infrastructure step (DNS + static host). Use GitHub Pages or
  Cloudflare R2 in production; the URL `https://tunapop.app/appcast.xml`
  is a placeholder.
- Automatic appcast updates from `release.yml` — Phase 15.x. v1
  releases append to `appcast.xml` manually.
- Sparkle's delta updates (binary diffs).
- App Store submission — separate sandbox rework required.
- Crash reporting via Sentry / Crashlytics — master spec opts out.
- Telemetry endpoints — master spec opts out.
- TestFlight-style beta channel.
